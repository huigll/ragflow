# 部署 Runbook: 192.168.12.120 (GPU 服务器)

维护者备忘。服务器: `carbit@192.168.12.120`(sudo),仓库在
`/mnt/ragflow-disk/home/yps/ragflow`(属主 `carit`,git 操作用
`sudo -u carit git -C <repo> -c safe.directory=<repo> ...`)。

## 关键事实(踩过的坑)

1. **服务器连不上 GitHub**。代码更新用 git bundle:
   本机 `git bundle create u.bundle main ^<服务器已有的sha>` → scp →
   服务器 `git fetch <bundle> main && git reset --hard FETCH_HEAD`(先 stash 本地改动)。
2. **GPU 依赖必须烧进镜像**(本仓库的 `Dockerfile.deploy` 最后一层)。
   RAGFlow 上游在 `DEVICE=gpu` 时运行时 `pip install torch`,在这台服务器上
   会卡死(pypi 不可达)或静默降级 CPU。**任何在容器里手工 pip install 的修复
   都会在容器 recreate 时丢失**——2026-07-05 就是这样复发的(此前 Codex 在
   容器内装过 libcublasLt.so.12,recreate 后蒸发)。
3. **CUDA 版本匹配**:onnxruntime-gpu 1.23.x 的 CUDA EP 需要 **CUDA 12**
   (`libcublasLt.so.12` + cuDNN 9)。默认 pypi 的 torch ≥2.12 带 CUDA 13 库,
   onnxruntime 用不了 → OCR 静默回退 CPU。必须用 cu12x 构建:
   `torch==2.9.1+cu128`(cu128 同时是 RTX 5090 D / Blackwell sm_120 的要求)。
   升级 torch 版本前先确认 onnxruntime 的 CUDA 要求。
4. `docker/.env` 里的 `{MYSQL_PW}` / `{ELASTIC_PW}` / `{OPENSEARCH_PW}`
   **是字面量真实密码**(整个栈用这些字符串初始化),不要"修复"它们。
   `RAGFLOW_SECRET_KEY` 必须保留,否则全部登录态失效。
5. 服务器上的未跟踪目录 `_deploy_wheels/ _deploy_graspologic/ _deploy_resource/`
   是离线构建资源,**不要 git clean**。
6. 120 专用的 TEI GPU、外部模型卷和单 worker 配置统一保存在
   `docker/docker-compose.120.yml`。不要再直接修改官方 Compose 文件。
7. 服务器 `docker/.env` 还必须保留以下非敏感运行参数:
   `DEVICE=gpu`、`WS=4`、`OCR_GPU_MEM_LIMIT_MB=12288`、
   `OCR_GPUMEM_ARENA_SHRINKAGE=1`、`MAX_CONCURRENT_CHUNK_BUILDERS=2` 和
   `TEI_IMAGE=ghcr.nju.edu.cn/huggingface/text-embeddings-inference:120-1.9`。
   密码和 `RAGFLOW_SECRET_KEY` 只保存在服务器,不得提交到 Git。

## 标准更新流程

```bash
# 0) 本机: 更新代码
git pull --rebase --autostash origin main   # 本地提交(含本文件、Dockerfile.deploy、entrypoint 补丁)会自动重放

# 1) 本机: 打完整 bundle 并交互式上传(服务器历史可能与本地 rebase 后分叉)
git bundle create /tmp/u.bundle main
scp /tmp/u.bundle carbit@192.168.12.120:/tmp/

# 2) 服务器: 更新代码(.env 是工作区改动,只恢复该文件)
G="sudo -u carit git -C /mnt/ragflow-disk/home/yps/ragflow -c safe.directory=/mnt/ragflow-disk/home/yps/ragflow"
$G stash push -m pre-deploy-env docker/.env
$G fetch /tmp/u.bundle main && $G reset --hard FETCH_HEAD
$G checkout stash@{0} -- docker/.env
$G stash drop stash@{0}
# 核对第4、7条的键,且 git status 只允许 docker/.env 有改动

# 3) 服务器: 构建(全程用缓存,~5-20 分钟)
cd /mnt/ragflow-disk/home/yps/ragflow
SHA=$($G rev-parse --short=12 HEAD)
sudo DOCKER_BUILDKIT=1 docker build --build-arg NEED_MIRROR=1 \
  -f Dockerfile.deploy -t ragflow-local:$SHA .

# 4) 服务器: 切镜像并重建容器(只动 ragflow-gpu;数据容器的卷不受影响)
sudo -u carit sed -i "s|^RAGFLOW_IMAGE=.*|RAGFLOW_IMAGE=ragflow-local:$SHA|" docker/.env
sudo docker compose -f docker/docker-compose.yml -f docker/docker-compose.120.yml \
  --project-directory docker up -d ragflow-gpu
```

## 部署后验证(缺一不可)

```bash
# a) Web 与数据
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:80/          # 200
sudo docker exec docker-mysql-1 mysql -uroot -p'{MYSQL_PW}' rag_flow \
  -N -e 'SELECT COUNT(*) FROM document;'                              # 与更新前一致
sudo docker inspect tei --format '{{.State.Health.Status}}'           # healthy

# b) GPU 三连(只看 found 1 gpus 不够!)
sudo docker exec docker-ragflow-gpu-1 /ragflow/.venv/bin/python3 \
  -c 'import torch; print(torch.cuda.is_available())'                 # True
# onnxruntime CUDA EP 必须真正激活(设置 entrypoint 里同款 LD_LIBRARY_PATH 后):
#   ort.InferenceSession(model, providers=["CUDAExecutionProvider"]).get_providers()
#   必须包含 CUDAExecutionProvider(回退 CPU 说明 CUDA 库版本不匹配)
# executor 日志(docker/ragflow-logs/task_executor_common_<hostname>_0.log):
#   "found 1 gpus" + 跑解析任务时 "load_model ... uses GPU"

# c) 更新前先备份数据库
sudo docker exec docker-mysql-1 mysqldump -uroot -p'{MYSQL_PW}' \
  --single-transaction rag_flow | gzip > ~/ragflow_mysql_backup_$(date +%Y%m%d_%H%M).sql.gz
```

## 回滚

改回 `docker/.env` 的 `RAGFLOW_IMAGE=` 为上一个 tag,然后执行:

```bash
sudo docker compose -f docker/docker-compose.yml -f docker/docker-compose.120.yml \
  --project-directory docker up -d ragflow-gpu
```

数据库迁移是增量的,旧镜像可直接跑新库。
